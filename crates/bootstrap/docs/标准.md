# bootstrap 标准文档

**包名**：`bootstrap` v0.3.2｜**角色**：L1 唯一组合根｜**状态**：L1 Internal Ready

## 能力与边界

- `Bootstrap`、`PlatformContext` 与 `AppContext` 组装 instrumentation、evidence、关停信号和正式 storage contracts。
- `ContractStoreSet` 只提供 `KeyValueStore` 与 `EventBus` 两个固定、可调用的 trait-object 槽位。
- 历史 `StoreSet` / `Bounded*` 仅为诊断替面；不得解释为正式 adapter 注册表。
- 不提供字符串 key、`Any`、`TypeId`、动态 `register`/`resolve` 或运行时服务发现。
- 泛型 `Repository<T, Id>` 由业务组合根按具体类型持有；多个槽位不表示跨资源事务。

## 可靠性与安全

- 正式依赖沿 builder → `PlatformContext` → `AppContext` 只读传递。
- `require_evidence` 的可恢复构建路径 fail-closed；关停 controller 仅触发一次。
- drain hook 按 LIFO 执行；完整异步资源编排仍由应用定义。
- 配置与密钥由上层注入，本 crate 不解析或记录凭据。

## 验收

```bash
cargo test -p bootstrap --all-targets
cargo clippy -p bootstrap --all-targets -- -D warnings
node scripts/storage-composition-conformance.mjs
```

固定摘要 Redis + NATS 实验只证明两个正式 contract 经 `AppContext` 可调用；不证明
Repository 容器、跨资源事务、服务发现或完整应用生命周期编排。
