# observex 标准

**Package / lib**：`observex` / `observex` v0.1.2 · `publish = false`

## 1. 定位

observex 提供 `contracts::Instrumentation` 的 tracing 实现，以及用于本地组合与测试的自定义有界
进程内 sink。它不实现 OpenTelemetry API/SDK、OTLP、远端持久化、采样或异步批处理。

## 2. 字段治理

- `TracingInstrumentation`、`PrefixedInstrumentation` 与 `ExportingInstrumentation` 发出的 `op`
  必须经过同一 `sanitize_op` 路径。
- 清理移除控制字符、trim、空值回落 `_`，并保证结果不超过 128 个 UTF-8 字节。
- 清理不是 PII/secret 检测，也不是 allowlist；调用方负责稳定词表、敏感性和基数治理。

## 3. 进程内 sink

- `InMemoryExporter` 的 span 与 metric 各自拥有独立容量；默认每类 1024，亦可显式配置。
- 容量不足时，单次同类批次全部拒绝，原缓冲不变，整批长度计入对应 dropped。
- `stats()` 在同一 mutex 临界区生成 buffered/flushed/dropped/shutdown 一致性快照；累计计数
  在 `usize` 范围内精确，溢出时饱和并设置 `counters_saturated`，此后相关数值只能作为下界。
- `flush()` 只清空进程内缓冲并累计 flushed，不代表写盘或远端确认。
- `shutdown()` 在同一临界区先 flush 计数再关闭；重复调用成功且不重复计数。
- shutdown 后 export/flush 返回 `ExportError::Shutdown`。

span 与 metric 使用两次独立 trait 调用，因此不承诺跨信号事务原子性。
容量限制事件数量；直接 exporter 调用方提供的 String/attributes 字节数不受该槽位数约束。

## 4. 失败与并发

- `ExportingInstrumentation` 先记录 inner，再同步调用 exporter；export 返回的 `Err` 与可展开
  （unwind）的 Rust panic 均被内化，不改变无返回值的业务记录调用，并累计诊断；
  `panic=abort` 不可捕获。
- 失败诊断中的 `unconfirmed_spans/metrics` 表示调用涉及但交付状态未知、wrapper 不重试；
  exporter 可能在失败前已有部分副作用，因此该数值不等同于实际 dropped。
- `TelemetryExporter` 实现必须快速返回，不得等待外部 I/O 或无界阻塞；违反合同的第三方实现
  仍会阻塞调用线程。本 crate 不提供 timeout 或异步线程隔离。
- `InMemoryExporter` 以 mutex 线性化 export/flush/shutdown，并从 poisoned mutex 恢复其状态。

## 5. 验收

```bash
cargo fmt --all --check
cargo test -p observex --all-targets
cargo clippy -p observex --all-targets -- -D warnings
cmp .agents/ssot/observex/spec/spec.md \

```

测试至少覆盖控制字符与多字节边界、容量 0/恰满/超限、批次原子拒绝、计数饱和状态、
并发容量守恒、exporter `Err` 隔离、shutdown 自动计数及幂等关闭。
