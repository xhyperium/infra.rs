# okxx 标准文档

**包名**：`okxx` v0.3.3 | **角色**：OKX 交易所适配器 | **层级**：Production-default

---

## 1. 需求标准

### 目标

okxx 提供 OKX 交易所的完整 REST+WS 接入能力，包括 4-header 签名认证、下单/撤单/查询、公共 WS 行情。

### 边界

- **包含**：4-header 认证（API Key/Passphrase/Timestamp/Signature）、REST 订单（place/cancel/query）、WS 行情（K线/深度/交易）、业务信封（event/arg/data 三层）
- **不包含**：私有 WS（账号/持仓）、衍生品/期权
- **依赖**：transportx、contracts、canonical、decimalx、base64、hmac

### 认证特点

OKX 使用 4-header REST 认证：

- `OK-ACCESS-KEY` — API Key
- `OK-ACCESS-SIGN` — HMAC-SHA256(timestamp + method + requestPath + body, secret)
- `OK-ACCESS-TIMESTAMP` — ISO 8601 UTC 时间戳
- `OK-ACCESS-PASSPHRASE` — API Passphrase

WS 需先发起 login 请求完成认证。

## 2. 设计标准

### 架构

```text
okxx
├── rest        — OkxRest（REST API 调用：4-header 签名）
├── ws          — OkxWs（WS 行情 + login 认证）
├── auth        — OkxAuth（4-header 签名器）
├── model       — 请求/响应 DTO + 业务信封
└── signer      — 签名中间件
```

### 数据流

```text
策略 → OkxRest.place_order(instId, tdMode, side, sz, px)
     → auth.sign(timestamp, method, path, body)
     → HTTP POST /api/v5/trade/order
     → 响应 → 业务信封 → OrderResult

WS: OkxWs.connect() → login(signature) → subscribe("tickers", "BTC-USDT")
```

### 业务信封

OKX 所有 REST 响应包裹在 `{ code, msg, data[] }` 信封中。
code = "0" 表示成功，非零值标识业务错误类型。

## 3. 编码标准

### 命名

- REST 方法按 OKX API 5.0 命名（place_order/cancel_order/amend_order）
- DTO 使用 snake_case，serde 映射到 OKX camelCase
- WS 事件按 OKX 频道命名（tickers/candles/books）

### 错误处理

- 业务信封 code 非零 → XError::Invalid（含 code + msg）
- HTTP 错误码 4xx/5xx → XError 映射
- WS login 失败 → 自动重试三次后报错

### 日志

- 下单/info!（不含密钥）
- WS 连接/warn!（含重连状态）
- 业务错误/warn!（含 code + msg）

## 6. 安全标准

### 密钥管理

- API Key + Secret + Passphrase 三要素通过配置注入
- 禁止日志输出任何认证要素
- 支持密钥脱敏（secret redaction）

### 签名

- `OK-ACCESS-SIGN` = Base64(HMAC-SHA256(timestamp + method + path + body, secret))
- 时间戳防重放（5 秒窗口）
- WS login 同样需要签名

### 环境隔离

- 模拟盘（Simulated）与实盘（Live）通过 baseURL 区分
- 模拟盘密钥与实盘密钥严格分离

### TLS

- 强制 HTTPS
- WS 使用 wss://

---

## 7. 可靠性标准

### 重试

- Transient 错误（5xx、429）自动重试
- 指数退避 + 抖动
- 最大重试次数：3

### WS 重连

- 断开自动重连
- 重连后需重新 login 恢复订阅
- 重连间隔：1s、3s、15s（递增）

### 幂等

- 下单使用 clOrdId（客户订单ID）实现幂等
- 同一 clOrdId 重复调用返回相同结果

## 8. 可维护性

- 按功能模块划分（rest/ws/auth/model）
- OKX API 文档链接内联
- 新增 API 端点只需添加 REST 方法
- 支持备用 REST 端点配置

## 9. 数据标准

### 数据源

- 行情：WS 推送（ticker/K线/深度/成交）
- 交易：REST 查询

### 精度

- 价格/数量精度由 instId 的 tickSz/lotSz 定义
- decimalx 管控金融运算

### 时区

- 所有时间戳 UTC
- REST 响应 ISO 8601
- WS 事件 UTC 毫秒

---

## 10. 风控标准

### 仓位管理

- 订单数量/价格须在 instId 可交易范围内
- 禁止超可用余额下单

### 限额控制

- OKX API 限流按秒计算（不同 endpoint 不同权重）
- 本地限流计数器
- 429 响应触发冷却

### 熔断

- 连续 3 次下单失败 → 暂停 30s
- WS 异常断开 → 暂停交易
- 模拟盘/实盘独立熔断

### 异常交易拦截

- 价格偏离局部市价超过阈值 → 告警
- 大额订单（超过配置限额）→ 拦截或二次确认

## 12. 上线标准

### 灰度策略

- 模拟盘投入运行 → 确认接口/行情正确
- 按交易对逐步开放（低流量→高流量）

### 监控指标

- 订单延迟 P50/P99
- WS 连接状态
- 业务错误码统计
- 限流余量

### 告警规则

- WS 断开 > 30s
- 下单失败率 > 5%
- 限流接近上限
- 业务错误码连续异常

### 回滚方案

- 切换备份交易所
- 关闭 OKX 交易（只读）
