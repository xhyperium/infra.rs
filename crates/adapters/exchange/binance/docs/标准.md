# binancex 标准文档

**包名**：`binancex` v0.3.2 | **角色**：币安交易所适配器 | **层级**：Production-default

---

## 1. 需求标准

### 目标

binancex ���供币安交易所的完整 REST+WS 接入能力，包括下单、撤单、查询、公共 WS 行情订阅。

### 边界

- **包含**：HMAC-SHA256 签名认证、REST 订单接口（place/cancel/query）、WS 行情（K线/深度/交易）
- **不包含**：私有 WS 数据流（账号/订单推送）、合约/期权产品
- **依赖**：transportx、contracts、canonical、decimalx、sha2、hmac

### 核心接口

| 符号 | 说明 |
|------|------|
| BinanceRest | REST 客户端（place_order/cancel_order/get_order） |
| BinanceWs | WS 行情客户端（subscribe_kline/subscribe_depth/subscribe_trade） |
| BinanceAuth | HMAC-SHA256 签名器 |

---

## 2. 设计标准

### 架构

```text
binancex
├── rest        — BinanceRest（REST API 调用）
├── ws          — BinanceWs（WS 行情订阅）
├── auth        — HMAC-SHA256 签名
├── model       — 请求/响应 DTO 定义
└── signer      — 请求签名中间件
```

### 数据流

```text
策略 → BinanceRest.place_order(symbol, side, quantity, price)
     → auth.sign(payload) → HTTP POST /api/v3/order
     → 响应解析 → OrderResult

WS: BinanceWs.connect() → subscribe_kline("BTCUSDT") → ws消息 → KlineEvent
```

## 3. 编码标准

### 命名

- REST 端点按 binance API 文档命名（place_order/cancel_order/get_order）
- DTO 使用 snake_case，serde 映射到 binance 的 camelCase
- WS 事件类型按 binance WebSocket 流命名

### 错误处理

- HTTP 错误码映射到 XError
- binance 业务错误码保留在错误上下文中
- WS 断开自动触发重连信号

### 日志

- 订单操作：info!（含订单ID但不含完整密钥）
- WS 连接/断开：warn!
- 限流响应：warn!（429 状态码）

### 注释

- 公开 API 完整 rustdoc
- 引用 binance API 文档链接
- 复杂签名逻辑添加中文注释

## 6. 安全标准

### 密钥管理

- API Key + Secret 通过配置注入，禁止硬编码
- 禁止日志中输出密钥（包括签名前的 payload）
- 密钥支持配置脱敏（secret redaction）

### 签名

- HMAC-SHA256 对请求参数排序后签名
- RECVWINDOW 限制请求有效期（默认 5000ms）

### TLS

- 强制 HTTPS（HTTP 请求被拒绝）
- WS 使用 wss://

### 依赖漏洞

- cargo deny 扫描依赖CVE
- 定期更新 reqwest/tokio-tungstenite

---

## 7. 可靠性标准

### 重试

- Transient 错误（5xx、429）自动重试
- 指数退避 + 抖动（通过 resiliencx）
- 最大重试次数：3

### WS 重连

- 连接断开自动重连
- 重连后自动恢复订阅
- 重连间隔：1s、5s、30s（递增）

### 幂等

- 下单使用 Binance 的新订单响应（orderId 唯一标识）
- 同一 clientOrderId 重复调用返回同一订单

## 8. 可维护性标准

### 可读性

- 模块按功能划分（rest/ws/auth/model）
- binance API 文档链接内联
- DTO 与 binance 响应字段名对应

### 可修改性

- 新增 API 端点只需添加 REST 方法
- WS 新增订阅类型只需添加订阅方法

### 可观测性

- 每次 REST 调用均有 tracing span
- WS 连接状态可通过 observex 监控

---

## 9. 数据标准

### 数据源

- 行情数据：WS 推送（K线1s-1月、深度、成交）
- 订单数据：REST 查询响应

### 精度

- 价格/数量以 binance 交易所精度为准（步长限制）
- decimalx 处理金融数据，禁止 f64 参与

### 时区

- 所有时间戳为 UTC
- WS 事件时间使用 Unix 毫秒
- REST 响应时间戳使用 Unix 毫秒

---

## 10. 风控标准

### 仓位管理

- 订单数量须校验在交易所允许范围内
- 禁止超过账户可用余额的下单

### 限额控制

- 遵守 binance API 限流规则（权重制）
- REST 请求前检查本地限流计数器
- 429 响应触发冷却

### 熔断

- 连续下单失败（3次）触发熔断
- 熔断后暂停交易 30s
- WS 断开异常触发暂停

### 异常交易拦截

- 订单价格偏离市场价超过阈值（可配置百分比）须告警
- 大额订单（超过配置限额）须二次确认

## 11. 回测标准

### 滑点模型

- 按市场深度模拟滑点（从 WS 深度快照计算）
- 滑点可配置（固定/比例/深度驱动）

### 手续费

- 按 binance 费率表计算（maker/taker）
- 支持 VIP 折扣配置
- BNB 抵扣手续费支持

### 未来函数防护

- 回测中使用 tick 级数据而非 K 线
- 禁止使用"未来"数据判定入场/出场
- 严格订单撮合时序

### 过拟合控制

- 样本内外数据比例 ≥ 7:3
- 参数敏感性分析
- 多市场交叉验证

## 12. 上线标准

### 灰度策略

- 先开通只读 API（查询+WS 订阅）
- 确认行情数据正确后开启下单
- 按交易对逐步开放（先低流量后高流量）

### 监控指标

- 订单延迟（P50/P99）
- WS 连接状态（已连/断开/重连次��）
- API 限流余量
- 错误率（按错误码分类）

### 告警规则

- WS 断开 > 30s → 告警
- 下单失败率 > 5% → 告警
- API 限流接近上限 → 告警
- 连续撤单失败 → 告警

### 回滚方案

- 切换至备份交易所
- 关闭该交易所交易（只读模式）
- 恢复上一已知良好版本
