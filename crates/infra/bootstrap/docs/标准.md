# bootstrap 标准文档

**Package / lib**：`bootstrap` / `bootstrap` v0.3.3 · `publish = false`
**角色**：L1 启动依赖组装（唯一组合根） · R3.1 豁免 / ADR-016

## 1. 范围

bootstrap 在进程启动时以类型安全方式组装：

- `PlatformContext` / `AppContext` 横切依赖；
- `StoreSet` 类型化适配器句柄；
- `ContractStoreSet` 正式 `KeyValueStore` / `EventBus` 固定 trait-object 槽位；
- 六个本地 `Bounded*` trait 的有界上下文；
- 唯一 shutdown guard、可克隆 signal 与同步 LIFO drain；
- instrumentation、可选 evidence 与组合错误映射。

不包含字符串 / `Any` / `TypeId` Service Locator、运行时服务发现、IoC 容器、
异步 runtime、分布式关停编排器或完整业务应用生命周期。

直接生产依赖仅为 workspace path package：`kernel`、`contracts`、`observex`、
`evidence`；固定摘要 Redis/NATS 组合实验另使用集中声明的第三方 dev-dependencies 与两个 adapter path dev-dependencies。

## 2. 模块与所有权标准

```text
bootstrap
├── lib         — Bootstrap、PlatformContext、AppContext、BootstrappedApp
├── bounded     — MarketDataContext / ExecutionContext
├── store_set   — StoreSet 类型化适配器接线
├── contract_store_set — 正式 KV/EventBus 固定槽位
├── drain       — AsyncDrain 同步快照与 LIFO 执行
├── traits      — 本地 Bounded* 与 evidence/instrumentation re-export
└── error       — BootstrapError → kernel::ErrorKind
```

- `ContractStoreSet` 必须保持固定 typed 槽位与只读访问器，不得演化为通用 Service Locator、Repository 容器或跨资源事务协调器。
- 未显式取走 guard 时，所有成功的公开 build 路径都必须把唯一 shutdown guard
  留在产物内。
- `graceful_shutdown(self)` 必须返回 `Result<Vec<DrainStepResult>, BootstrapError>`，
  先触发 signal，再 drain；单步 hook `Err` 不得短路后续 hook。
- ownerless context 只有在 signal 已由外部 owner 触发时才可 graceful drain；否则返回
  `MissingDependency("shutdown_guard")` 且不执行 hook。错误消费 context，不能重试。
- `trigger_shutdown(self)` 只触发 signal，不得隐式执行 drain。
- `into_parts` 是兼容型显式转移：guard 进入 `ShutdownController`，拆出的
  `AppContext` 不再有触发所有权，但仍可 drain。

## 3. Drain 并发、阻塞与失败边界

`AsyncDrain` 保留历史类型名，但当前不是异步执行器：hook 为同步
`FnOnce() -> XResult<()>`，在调用线程执行。

- `register` 与 drain 快照在同一 mutex 上线性化；
- 快照前完成的注册进入本批，快照后的注册进入下一批；
- hook 在锁外执行，不承诺不同批 hook 串行或跨批全局 LIFO；
- 返回 `Err` 会被记录并继续；hook panic 不隔离，会中断当前调用；
- 任一阻塞 hook 都可能永久阻塞 drain。
- 注册 mutex 中毒按依赖不可用返回并保留下层原因，不按调用方配置错误处理。

因此 bootstrap **不承诺** drain 的 timeout、取消或 panic 安全。需要这些能力的
调用方必须在每个 hook 内提供 deadline、合作式取消、join 与升级策略。

bootstrap 自有用户可见错误必须使用简体中文。`BootstrapError` Display 与转换后的
`XError` context 使用统一中文前缀；DependencyUnavailable 顶层 Display / context 不得内插任意
下层 source 文本，结构化 `#[source]` 链必须保留。re-export 类型及
opaque source/hook context 的语言由其定义 crate 负责，不得为翻译而吞掉原因链。

## 4. 验证标准

最低门禁：

```text
cargo fmt -p bootstrap -- --check
cargo test -p bootstrap --all-targets
cargo clippy -p bootstrap --all-targets -- -D warnings
node scripts/storage-composition-conformance.mjs
node scripts/quality-gates/cov-gate-100.mjs -p bootstrap --filter crates/infra/bootstrap/src
```

测试必须覆盖四条 build 路径的关停所有权、signal-before-drain、ownerless fail-closed、
批内 LIFO、错误后继续与完整结果、`into_parts` 转移、旧 trigger-only 行为、
register/drain 快照线性化、锁中毒错误映射与正式 KV/EventBus 组合实验。测试通过只证明上述进程内合同，不等于 package stable、
真实应用生命周期或生产关停 SLA 已闭合。
