# DDR-007: Squash Merge 作为唯一合并略

**日期**: 2026-07-21
**状态**: 已接受

## 上下文

PR 合并策略影响 `main` 分支的历史可读性和回滚复杂度。常见选项包括 merge commit、squash merge 和 rebase merge。

## 选项

| 选项 | 优点 | 缺点 |
| ------ | ------ | ------ |
| Merge commit | 保留完整开发历史 | main 分支噪音大，fixup/调试提交混入 |
| Rebase merge | 线性历史，无合并提交 | 丢失 PR 边界信息 |
| **Squash merge** | 线性历史 + 保留 PR 边界 | 丢失分支内细粒度提交历史 |

## 决策

采用 **squash merge** 作为唯一合并策略。每个 PR 合并为 `main` 上的单个提交。

## 后果

- **正面**: `main` 分支历史简洁线性，易于 `git bisect`
- **正面**: 每个提交对应一个完整的功能/修复单元
- **正面**: 合并后自动清理分支
- **负面**: 丢失分支内部的渐进式提交历史
- **负面**: 需确保 PR 描述完整，补偿 squash 丢失的上下文
