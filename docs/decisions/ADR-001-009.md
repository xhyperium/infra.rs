# Architecture Decision Records (ADR)

本文档记录 infra.rs 项目中的关键架构决策，每个决策按 DDR（Design Decision Record）编号，包含上下文、选项分析、决策及后果。

---

## DDR-001: Rust 2024 Edition + MSRV 1.85

**日期**: 2026-07-21
**状态**: 已接受

### 上下文

项目初始化时需选定 Rust 工具链版本策略。Rust 2024 版号是最新稳定版号（2025 年 2 月发布），带来了多项语言改进：`unsafe` 作用域收紧、`impl Trait` 语法简化、`gen` 块预览等。

### 选项

| 选项 | 优点 | 缺点 |
|------|------|------|
| Edition 2021 + MSRV 1.70 | 最大兼容性，CI 镜像丰富 | 无法使用新版语言特性；与下游 xhyper.rs 不一致 |
| Edition 2021 + MSRV 1.85 | 更好的工具链支持 | 版号与 MSRV 版本号割裂 |
| **Edition 2024 + MSRV 1.85** | 最新语言特性；与下游 xhyper.rs 对齐 | 部分旧工具可能未适配 |

### 决策

采用 **Rust 2024 Edition + MSRV 1.85**。

### 后果

- **正面**: 与 `xhyperium/xhyper.rs` 保持工具链一致，减少跨仓库摩擦
- **正面**: 可使用新版号特性（如改进的 unsafe 语义）
- **负面**: 需要 CI 中专门配置 MSRV 验证 Job
- **负面**: 限制了部分旧 Rust 工具链用户的兼容性

## DDR-002: thiserror v2 作为错误派生方案

**日期**: 2026-07-21
**状态**: 已接受

### 上下文

infra-core 需要一个类型安全的错误处理方案。Rust 生态中主要有三种方式：`std::error::Error` 手动实现、`thiserror` 派生宏、`anyhow` 动态错误。

### 选项

| 选项 | 优点 | 缺点 |
|------|------|------|
| 手动实现 `std::error::Error` | 无外部依赖 | 样板代码多，易出错 |
| **thiserror v2** | 派生宏减少样板；类型安全；社区标准 | 增加编译时间，额外依赖 |
| anyhow | 极简 API，适合应用层 | 类型擦除丢失错误信息；不适合库 |

### 决策

采用 **thiserror v2** 作为 `infra-core` 的错误派生方案。

### 后果

- **正面**: `#[derive(Error)]` 一行宏替代 20+ 行手动实现
- **正面**: `#[from]` 属性自动实现 `From` 转换，支持 `?` 传播
- **正面**: `#[error("...")] ` 统一 `Display` 格式
- **负面**: 增加编译依赖（thiserror + thiserror-impl + syn 等）

## DDR-003: serde 手动实现 Error 序列化

**日期**: 2026-07-21
**状态**: 已接受

### 上下文

`infra_core::Error` 需要在网络边界（HTTP API、IPC、日志导出）传输。`serde::Serialize`/`Deserialize` 是 Rust 生态标准。对于 `std::io::Error`，简单的 derive 实现会丢失 `source()` 链。

### 选项

| 选项 | 优点 | 缺点 |
|------|------|------|
| 仅 derive Serialize/Deserialize | 简单，零手写代码 | IO error 链丢失，`error_kind` 无法保留 |
| serde 的 remote derive | 保持外部类型序列化 | 不适用于包含动态 source() 链的 io::Error |
| **手动实现** | 完整保留 error_kind 和 source() 链 | 代码量较大（约 150 行） |

### 决策

为 `Error` 枚举**手动实现 `serde::Serialize` 和 `serde::Deserialize`**，通过 `ChainNode` 辅助结构体完整保留 IO 错误链。

### 序列化格式

```json
// Io 变体
{"kind": "Io", "error_kind": "NotFound", "chain": ["top level", "source 1", "root"]}

// 其他变体
{"kind": "Config", "message": "missing field host"}
```

### 后果

- **正面**: 完整的跨边界错误传播能力
- **正面**: 反序列化后可重建完整的 `source()` 调用链
- **负面**: 约 150 行手动实现代码需维护
- **负面**: 新增 `io::ErrorKind` 变体时需更新 match 臂

## DDR-004: 中文优先注释与文档

**日期**: 2026-07-21
**状态**: 已接受

### 上下文

项目团队以中文为母语。文档和注释的语言选择影响开发效率、代码审查速度和新成员上手成本。

### 选项

| 选项 | 优点 | 缺点 |
|------|------|------|
| 全英文 | 国际化标准，开源社区惯用 | 团队表达精度下降；审查效率降低 |
| **中文优先** | 团队认知负担最低；表达精确 | 非中文贡献者门槛高；工具链支持需验证 |
| 中英混用 | 灵活 | 不一致性高，风格难以统一 |

### 决策

**中文优先**: 代码注释、项目文档、用户可见错误信息使用中文。技术术语保留英文。标识符使用英文（Rust 惯例）。英文技术文档遵循 ASD-STE100 规范。LICENSE 保留原文。

### 后果

- **正面**: 团队内部沟通和代码审查效率显著提升
- **正面**: 认知负担降低，减少翻译损耗
- **负面**: 非中文社区成员贡献门槛提高
- **负面**: 部分文档工具（拼写检查）需额外配置

## DDR-005: 统一 target-dir 到 .cargo/target/

**日期**: 2026-07-21
**状态**: 已接受

### 上下文

Rust 默认将构建产物放在 `target/`。在 workspace 根目录运行 `cargo build` 时产物在 `./target/`，但在子目录运行可能产生额外 `target/` 目录。CI 缓存策略和本地清理需一致性。

### 选项

| 选项 | 优点 | 缺点 |
|------|------|------|
| 默认 `target/` | 零配置 | 多位置 target 导致缓存碎片 |
| 环境变量 `CARGO_TARGET_DIR` | 灵活，但需每个环境设置 | 容易遗忘配置 |
| **.cargo/config.toml 统一设置** | 仓库内声明，CI 和本地一致 | 对习惯默认 target/ 的开发者有学习成本 |

### 决策

通过 `.cargo/config.toml` 设置 `target-dir = ".cargo/target/"`，在仓库内统一所有构建产物路径。

### 后果

- **正面**: CI 和本地开发使用相同路径，Swatinem/rust-cache 命中率更高
- **正面**: `.cargo/` 整体 gitignored，清理只删一个目录
- **正面**: 工具缓存（advisory-db、mutants）也可放在 `.cargo/cache/` 下
- **负面**: 开发者需习惯构建产物不在根目录 `target/`

## DDR-006: Beads (bd) 作为任务跟踪系统

**日期**: 2026-07-21
**状态**: 已接受

### 上下文

infra.rs 是多 AI 代理（Claude Code、Codex、Copilot）协作的元仓库。传统 GitHub Issues 无法被 AI 代理以结构化方式读写。需要一种跨模型、可编程的任务跟踪方案。

### 选项

| 选项 | 优点 | 缺点 |
|------|------|------|
| GitHub Issues | 无需额外工具；与 PR 集成 | AI 编程读写受限；缺少结构化依赖 |
| 纯 Markdown TODO | 极简 | 无依赖跟踪；多代理冲突风险 |
| **Beads (bd)** | CLI 可编程；Dolt DB 结构化存储；跨模型共享 | 额外工具学习成本；需本地 Dolt 运行环境 |

### 决策

采用 **Beads（`bd` CLI）** 作为任务跟踪系统。本地使用 Dolt 数据库，通过 `refs/dolt/data` 在 git remote 上同步。`.beads/` 目录 gitignored，不进入版本库。

### 后果

- **正面**: `bd create/update/close/dep` 命令行全流程覆盖
- **正面**: 结构化依赖跟踪（`bd dep add`），支持 blocking/blocked 关系
- **正面**: 跨模型共享（Claude Code 读写，Codex 读写，Copilot 只读）
- **负面**: 团队成员需额外安装和学习 `bd` 工具
- **负面**: Dolt 数据库管理与 git 工作流的集成需额外理解

## DDR-007: Squash Merge 作为唯一合并策略

**日期**: 2026-07-21
**状态**: 已接受

### 上下文

PR 合并策略影响 `main` 分支的历史可读性和回滚复杂度。常见选项包括 merge commit、squash merge 和 rebase merge。

### 选项

| 选项 | 优点 | 缺点 |
|------|------|------|
| Merge commit | 保留完整开发历史 | main 分支噪音大，fixup/调试提交混入 |
| Rebase merge | 线性历史，无合并提交 | 丢失 PR 边界信息 |
| **Squash merge** | 线性历史 + 保留 PR 边界 | 丢失分支内细粒度提交历史 |

### 决策

采用 **squash merge** 作为唯一合并策略。每个 PR 合并为 `main` 上的单个提交。

### 后果

- **正面**: `main` 分支历史简洁线性，易于 `git bisect`
- **正面**: 每个提交对应一个完整的功能/修复单元
- **正面**: 合并后自动清理分支
- **负面**: 丢失分支内部的渐进式提交历史
- **负面**: 需确保 PR 描述完整，补偿 squash 丢失的上下文

## DDR-008: 12 钩子 + 46 技能 + 28 Codex 代理的 AI 治理分层

**日期**: 2026-07-21
**状态**: 已接受

### 上下文

infra.rs 的核心使命之一是作为 AI 编码助手的治理中枢。需要定义多个代理系统（Claude Code、Codex、Copilot）的协作规则，避免冲突和配置漂移。

### 决策

采用三层 AI 治理架构：

- **Claude Code**: 主执行代理 — 12 个生命周期钩子控制会话全流程
- **Codex**: 编排派工代理 — 28 个专业代理处理不同类型任务
- **Copilot**: 补充建议代理 — 自动管理，不参与写入

所有技能定义以 `.claude/skills/` 为 SSOT。`.codex/.agents/skills/` 为自动投影，禁止手工编辑。

### 钩子生命周期分层

```
SessionStart: session-context + bd prime（上下文注入）
PreToolUse:   pre-tool-check + edit-guard + count-guard（操作前校验）
PostToolUse:  post-tool-check + edit-guard-reset + link-check（操作后检查）
PreCompact:   pre-compact（压缩前状态保留）
Stop:         session-review + version-guard + branch-protect（会话结束审查）
```

### 后果

- **正面**: Claude Code 会话全生命周期可审计
- **正面**: SSOT 防止技能定义在多代理间分叉
- **正面**: 分层设计使每个代理的能力边界清晰
- **负面**: 配置文件数量多（86+ 个），维护成本高
- **负面**: 新成员理解全貌需要时间

## DDR-009: 统一使用 taiki-e/install-action@v2 安装 CI 工具

**日期**: 2026-07-21
**状态**: 已接受

### 上下文

CI 工作流需要安装多个 Rust 生态工具：`cargo-nextest`、`cargo-llvm-cov`、`cargo-deny`、`cargo-audit`、`taplo-cli`。这些工具的安装方式各异（GitHub Action 封装、cargo install、二进制下载），不一致会增加维护负担。

### 选项

| 选项 | 优点 | 缺点 |
|------|------|------|
| 各自使用官方 Action / cargo install | 可能有更针对性的配置 | 安装方式碎片化；版本更新需逐一处理 |
| 统一 script 脚本 | 可控性强 | 需维护额外脚本 |
| **taiki-e/install-action@v2** | 统一接口；二进制缓存快速；支持 50+ 工具 | 依赖第三方 Action 的可用性 |

### 决策

CI 中所有 Rust 工具安装统一使用 **`taiki-e/install-action@v2`**。与 `Swatinem/rust-cache@v2` 配合使用，首次下载后缓存二进制文件。

### 使用清单

| 工具 | 用途 | 工作流 |
|------|------|--------|
| `cargo-nextest` | 测试运行器 | CI (Rust) |
| `cargo-llvm-cov` | 代码覆盖率 | CI (Rust) |
| `cargo-deny` | 依赖安全审计 | 安全 |
| `cargo-audit` | 漏洞扫描 | 安全 |
| `taplo-cli` | TOML 格式检查 | 校验 |

### 后果

- **正面**: 统一安装接口，新增工具时只需改一行
- **正面**: 二进制缓存加速 CI，首次安装后数秒内完成
- **正面**: 版本自动跟随 latest，无需手动更新
- **负面**: 依赖单点（taiki-e/install-action）
- **负面**: 某些工具可能需要特定功能开关，可通过 `with.tool` 指定
