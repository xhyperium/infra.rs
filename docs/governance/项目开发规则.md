# 项目开发规则

本文档汇总 infra.rs 仓库的工程开发与文档管理规则。

---

## §1 文档规范

### 报告保存路径

所有审查报告、审计报告、生成报告统一保存至：

```text
docs/report/{YYYY-MM-DD}/
```

- 日期格式：ISO 8601（四位年-两位月-两位日）
- 每个日期目录必须包含 `README.md` 索引
- 细则见：[文档组织约定.md](文档组织约定.md)

### 编码与语言

人类可读文本强制简体中文，UTF-8（无 BOM），LF 换行。细则见：[编码与语言约定.md](编码与语言约定.md)。

## §2 工程规范

### 版本策略

crate 独立版本号，每次交付 PATCH +1。细则见：[VERSIONING.md](VERSIONING.md)。

### Git Worktree

所有活跃开发在 `.worktrees/<branch-name>` 内进行，禁止在 `main` 上直接开发。细则见：[worktree-policy.md](worktree-policy.md)。

### 提交规范

遵循 [Conventional Commits](https://www.conventionalcommits.org/) 格式，标题使用中文简述。

仓库已激活 `.gitmessage` 模板（`git config commit.template`），执行不带 `-m` 的 `git commit` 时自动加载。模板详见：[commit-template.md](commit-template.md)。

#### 格式

```text
<type>[!][(scope)]: <中文简述>
```

| 字段 | 必选 | 取值 |
|------|------|------|
| `type` | 是 | `feat` `fix` `chore` `docs` `refactor` `test` `perf` `ci` `style` `build` |
| `!` | 否 | 破坏性变更标记 |
| `scope` | 否 | `kernel` `testkit` `configx` `observex` `resiliencx` `bootstrap` 等 |
| `简述` | 是 | 一句话中文描述，不加句号 |

#### 示例

##### 功能提交

```text
feat(kernel): 新增生命周期 on_stop 回调

组件销毁前触发 on_stop，允许外部在清理阶段注入收尾逻辑。
```

##### 修复提交

```text
fix(schedulex): 修复任务登记表重复分配

两个并发 caller 在 claim_task 时可能分配到同一个 ID，
原因是 AtomicCounter 的 compare_exchange 未正确处理弱序失败。
```

##### 文档提交

```text
docs(governance): 新增文档组织约定

规定报告保存路径为 docs/report/{YYYY-MM-DD}/。
```

##### 工具链/杂务提交

```text
chore(deps): 迁移剩余第三方依赖至 workspace 集中管理
ci(merge-queue): 为 required workflow 添加 merge_group 触发器
```

##### 破坏性变更

```text
feat!(contracts): Exchange trait 新增 validate_order 方法

BREAKING CHANGE: 所有 Exchange 实现者必须实现 validate_order。
```

### 存储适配器配置

clickhousex / kafkax / natsx / ossx / postgresx / redisx / taosx 的环境变量规范与本地数据库创建指南。细则见：[storage-adapter-config.md](storage-adapter-config.md)。

### 优先复用内部库（DEV-001）

> 新增功能前，必须先检索内部库；已有实现可���接复用的，禁止重新实现。

| 条款 | 内容 |
|------|------|
| 开发前 | 检索内部 crate / 模块索引，确认是否已有相同或相��能力 |
| 可复用 | 直接依赖内部��，不得以「略有差异」为由重写 |
| 需扩展 | 优先向内部库提交 PR / 扩展接口，而非另起新模块 |
| 确无对应 | 新建模块时需注明检索结果与不可替代理由 |

**例外条件**（满足其一可新建）：
- 内部库性能不满足量化场景（需附 benchmark）
- 内部库接口无法适配且改造成本高于重写
- 内部库已废弃或无人维护

**违规处理**：
- Code Review 阶段发现重复实现 → 驳回，要求替换为内部库
- 合并后发现 → 记录技术债，限期迁移

### 代码复用��则（DEV-002）

> 相同逻辑只实现一次；可复用代码必须下沉为公共模块。

| 条款 | 内容 |
|------|------|
| 重复检测 | 相似逻辑出现 ≥2 次，必须抽象为公共函数 / 宏 / trait |
| 复用优先级 | 内部公共库 > 项目内公共模块 > 业务模块私有实现 |
| 禁止内联复制 | 不得通过复制粘贴绕过复用 |
| 接口稳定 | 公共接口变更需评估所有调用方影响 |

**例外条件**：
- 业务逻辑表面相似但语义不��，强行抽象会导致耦合
- 性能敏感路径，复用会引入不可接受开销��需附 benchmark）

### 独立复用模块规则（DEV-003）

> 复用模块必须独立、可测试、可单独发布，禁止依赖具体业务。

| 条款 | 内容 |
|------|------|
| 零业务依赖 | 不得引用上层业务 crate / 模块 |
| 独立编译 | 可单独 `cargo build` / `cargo test` |
| 接口明确 | 通过 `pub` API 暴露能力，内部实现对外不可见 |
| 版本管理 | 接口变更需遵循语义化版本 |
| 文档完备 | 提供用途、示例、约束说明 |

**禁止事项**：
- 禁止在复用模块中硬编码业务配置
- 禁止反向依赖调用方
- 禁止包含不可复用的临时逻辑

## §3 质量门禁

提交前必须通过：

```bash
cargo fmt --all --check
cargo clippy --workspace --all-features --all-targets -- -D warnings
cargo test --workspace
node scripts/quality-gates/check-workspace-deps.mjs
```
